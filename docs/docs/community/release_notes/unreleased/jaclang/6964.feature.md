- **Native: flat `import json; json.dumps(...)` form for bundled na_stdlib modules (#6964)**: Besides `import from json { dumps }`, the native (na) backend now accepts a flat `import json;` (or `zlib` / `datetime` / `hashlib`) followed by `module.func(...)` member calls, matching the Python-shaped ergonomics most real I/O tools use. The capability gate admits a flat **single-segment** import whose name `resolve_native_module` finds as a bundled (or project-local) `.na.jac`; `BoundaryAnalysisPass` enumerates that module's public top-level symbols and creates one NATIVE binding per symbol (driving the existing cross-module `link_in`); and a `json.dumps(x)` member call is rewritten to the linked free call `dumps(x)`, reusing the whole import-from machinery end to end. A dotted `import os.path;` stays rejected (use the `import from` form) so it cannot shadow the Mechanism-A `import os` member intercepts, which are untouched.
- **Native stdlib: bundled pure-Jac `hashlib` (SHA-256) (#6964)**: Adds `na_stdlib/hashlib.na.jac`, a Mechanism-B (pure-Jac, FIPS 180-4) SHA-256 over 32-bit integer arithmetic -- portable to every native target with no system library. `sha256()` returns a CPython-shaped hash object with `update(data)` / `hexdigest()` / `digest()`; the digest is pinned byte-for-byte against the FIPS / CPython vectors (empty, single-block, multi-block, and incremental `update`) by the new tri-backend `prim_hashlib` equivalence fixture. The round constants live in a function-local list rather than a module `glob`, so the module needs no `__jac_glob_init` ordering and works from an entry-less consumer.
