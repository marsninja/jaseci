- **Fix: native `datetime` now matches CPython for pre-1970 timestamps (#6954 follow-up)**: The bundled `na_stdlib/datetime.na.jac` computed the day split (`total // 86400`) and `weekday()` (`(epoch_day + 3) % 7`) with raw `//` and `%`. On the native (na) backend integer `//` lowers to LLVM `sdiv` (truncates toward zero) and `%` to `srem` (sign follows the dividend), whereas CPython floors both, so for any negative (pre-1970) POSIX timestamp the na backend silently diverged from the sv backend across day/hour/minute/second and weekday (e.g. `fromtimestamp(-86401.0)` gave `1969-12-31` on na vs CPython's `1969-12-30`). The module now uses floor-correct `_fdiv`/`_fmod` helpers for the day split and weekday, keeping the two backends congruent across the whole timestamp domain; positive timestamps are unaffected. A negative timestamp is pinned in the `prim_datetime` equivalence fixture as a regression guard.
- **Fix: static-field global lowering no longer hides real codegen errors**: `_register_static_fields` created each `static has` module global inside a bare `try { ... } except Exception { continue }` meant only to skip a name already lowered on a re-visited module. That also swallowed genuine codegen failures (bad type, malformed name) from `ir.GlobalVariable()` as if they were benign duplicates. It now checks the llvm module's own global table for the name and creates the global unconditionally otherwise, so only an actual duplicate is skipped and any other error surfaces normally.
