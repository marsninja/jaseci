- **Feature: bundled native `base64` (#6978 Phase 3)**: `import from base64 { ... }` now resolves on the native (`nacompile`/JIT) pathway to a bundled pure-Jac (Mechanism B) module (`na_stdlib/base64.na.jac`) with no FFI floor and no dependencies, so it links on every native target. The RFC 4648 codecs `b16encode`/`b16decode`, `b32encode`/`b32decode`, `b64encode`/`b64decode` (with `altchars` and `validate`), `standard_b64encode`/`standard_b64decode`, `urlsafe_b64encode`/`urlsafe_b64decode`, and the RFC 1924 `b85encode`/`b85decode` (the alphabet CPython's `base64.b85encode` uses) produce byte-identical output to CPython across all 256 byte values and round-trip its output. Decoding matches CPython 3.14 semantics: the default `b64decode` discards non-alphabet bytes (newline-wrapped MIME/PEM input decodes) and applies 3.14's end-of-input padding rules, `validate=True` implements strict mode with CPython's error set, `urlsafe_b64decode` accepts both the `+/` and `-_` alphabets, `b16decode`/`b32decode` take `casefold` (and `b32decode` `map01`) with CPython's length/pad-count validation, and `b85decode` reports CPython's absolute error positions and 32-bit overflow. Errors raise `ValueError` with CPython's message text (`binascii.Error` subclasses `ValueError`). SCOPE: the CPython `None` sentinels for `altchars`/`map01` are `b""` here, and the Ascii85 (`a85encode`/`a85decode`) variant is a follow-up. Pinned sv<->na congruent by the `prim_base64` equivalence fixture: digests and lengths of each encoding over ASCII and full 0-255 byte inputs, decode round-trips across every padding boundary, empty inputs, whitespace/junk-discard and strict-mode cases, and the error paths of every codec.
