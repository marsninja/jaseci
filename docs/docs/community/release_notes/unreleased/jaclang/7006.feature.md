- **Feature: bundled native `uuid` (v4) on the libcrypto CSPRNG floor (#6978 Phase 1)**: `import uuid` now resolves on the native (`nacompile`/JIT) pathway to a bundled `na_stdlib` Mechanism-F module (`uuid.na.jac`) over the same OpenSSL `RAND_bytes` floor (`_csprng_native.na.jac`) that `secrets` uses. `uuid4()` draws 16 CSPRNG bytes and renders the 32-char `hex`, applying the v4 version (byte 6 high nibble) and RFC-4122 variant (byte 8) bits during the render (native `bytes` are immutable, so the bits are applied at render time); `UUID.hex`, `str(UUID)` (the canonical `8-4-4-4-12` form, via native `__str__` dispatch from #7005), and `UUID.version` match CPython's API, and an `nacompile`d binary statically links `libcrypto.a` and runs self-contained. A new tri-backend equivalence test (`prim_uuid`) pins the native and CPython implementations congruent on the deterministic v4 structure of the random output (hex/str lengths, hyphen layout, version nibble, RFC-4122 variant nibble, `version == 4`, and that two draws differ). The `.bytes`/`.int`/`.urn` attributes (which need a constructed `bytes`) are follow-ups.
