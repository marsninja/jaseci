- **Fix: native `<enum> | None` return types lower to i64 instead of `void` (#6971)**: A native function declared `-> Color | None` collapsed its return type to `void`, so any caller that bound the result (`c = pick(...)`) could not lower (it failed LLVM verification with `value doesn't match function result type 'void'`, and the binding raised the E5090 void-binding diagnostic). When the type checker has not stamped a `.type` on the annotation, `_resolve_jac_type` falls back to the AST-based `_resolve_primitive_name`, which walks the union's sub-nodes and returns the first that resolves to a type; for `int | None` the `int` operand resolved first, but for a member the primitive table does not know (an enum or user struct) the walk fell through to the bare `None` operand, which resolves to `VoidType`. `_resolve_primitive_name` now peels a `T | None` union explicitly: it skips the `None` operand(s) and resolves the value member, deferring to the existing struct / i64 fallback for a non-primitive member, so `Color | None` lands on the same i64 + enum-marshalling path as a plain `-> Color` (the marshaller maps the ordinal back to the enum member, with the out-of-range sentinel mapping to `None`). This mirrors the `_lower_type` union branch (the TypeBase path) so both resolvers agree; plain `-> None` returns are unaffected. As a result the bootstrap lexer no longer has any void-binding demotions: its `lookup_keyword(text) -> TokenKind | None` now lowers across the module boundary.
