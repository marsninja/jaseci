- **Fix: native bare-`tuple`-annotated tuple-literal assignment no longer crashes codegen (#6964)**: `t: tuple = (10, 20);` aborted the LLVM IR builder with `cannot store {i64, i64} to i64*: mismatching types`. A bare `tuple` annotation carries no element layout, so `_resolve_jac_type` fell back to the `i64` default, which cannot hold the value-struct (`{i64, i64}`, `{i8*, i64}`, ...) the RHS literal already lowered to. The new-variable assignment path now adopts that struct type directly when the annotation is a bare `tuple` and the value is a literal struct -- mirroring the existing bare-collection (`list`/`dict`/`set`) inference. Parameterized `tuple[int, int]` and the un-annotated form already lowered correctly and are unchanged; the fix covers both homogeneous and heterogeneous element types.
