- **Fix: a bridged `na {}` function called from `sv` no longer aborts when it reads an imported na module's static field**: A regular `.jac` module that calls a native (`na {}`) function from its `sv` side (e.g. a `with entry`) runs by importing the module and `exec()`-ing its bytecode, with the native JIT engine injected by `meta_importer`. Unlike the AOT `jac_entry` startup preamble and the bootstrap `native_accel` path, `meta_importer` never ran the module's `__jac_glob_init` (nor its imported na deps' `__jac_glob_init_<dep>`) before executing the body. So a bridged native call that read a module-level global -- or, transitively, an imported na module's `static has` class attribute such as the bundled `datetime` module's `timezone.utc` -- dereferenced an uninitialized null and aborted the process (SIGABRT). `meta_importer` now runs the existing `init_native_globals` helper (which initializes the module's own globals plus every module in `InteropManifest.native_transitive_sources`, in dependency order) immediately before executing the module body, matching the AOT and acceleration paths. Pure-native execution (`jac nacompile`, or an all-native `jac run` of a `.na.jac`) was already correct. New end-to-end `na`<->`sv` bridge equivalence tests (`tests/compiler/passes/native/test_na_sv_bridge.jac`) pin the bundled `json` / `zlib` / `os` / `datetime` modules and a minimal imported-static-field repro congruent across both backends through the live bridge.
