- **Feature: the Jac-native desktop host runs on the fused single-binary runtime**: instead of binding the build machine's `libpython` by soname (`INSTSONAME` + an `LD_LIBRARY_PATH`/RUNPATH to its `LIBDIR`), a desktop app now brings up the **same fused runtime the `jac` CLI ships** -- the pinned, bundled CPython + jaclang carried in the binary's trailer payload. The CPython bring-up (materialize the payload → set a hermetic env → `dlopen` the bundled libpython → pin the program name → resolve the C-API) is unified in `launcher/embed.zig`, shared by the headless CLI launcher and the desktop host. The desktop host reaches it through a small C-ABI shim, `libjacpyembed` (`launcher/pyembed.zig`): the generated `host.na.jac` imports it by logical name (`import from jacpyembed`, resolved as a `$ORIGIN` sibling -- no `patchelf` for libpython), and `jac_engine_boot()` materializes the trailer and initializes the interpreter. The build appends this `jac`'s `[payload][trailer]` onto the compiled host, so it carries the bundled runtime and **auto-dedupes to the CLI's `rt/<hash>` cache** (identical payload hash). The shim ships inside the payload (`--pyembed`) and is staged `$ORIGIN`-adjacent at build time. Net effect: no build-machine libpython, no `INSTSONAME`, no `LD_LIBRARY_PATH`, and no per-build "bake the builder's site-packages into the binary" step -- the desktop app is the jac engine plus a window.
- **Feature: desktop bootstrap modules are imported from the fused runtime**: `oauth_broker`, `inprocess_dispatch`, and the desktop plugins are no longer `jac2py`-transpiled and staged as `.py` beside the binary -- the embedded interpreter imports them straight from the bundled jaclang tree (the plugins via their runtime dir on `sys.path`, with `import registry` cascading to every `built_in.*` plugin). Removes three staging passes and their `jac2py`/`jac0` subprocesses; only the app's own `sv` source tree and `dist/` bundle are still staged.
- **Feature: `na`/LLVM is first-class in the desktop runtime**: because the desktop app bundles the same payload the CLI ships -- which carries the LLVMPY shim and the native compiler -- an app's runtime can drive `jac nacompile` / `na` exactly like the CLI.
- **Feature: In-process `sv` dispatch in the Jac-native desktop host** (Phase J4), now running on the fused runtime: `cl` → `sv` walker/function calls on the desktop target run through the embedded CPython in-process instead of an HTTP POST to a remote backend. `inprocess_dispatch` boots the jac-scale ASGI app over a Starlette `TestClient` (no socket) and serves as the native `window.__jac_invoke` trampoline with the same `{kind, name, node, fields, token}` request contract and `{status, ok, body}` reply semantics as the web API. Dispatch runs on a single serialized worker thread off the GTK UI thread so a slow walker never freezes the window, with a configurable per-call timeout (`NATIVE_DISPATCH_TIMEOUT`, non-preemptive) and a bounded queue (`NATIVE_DISPATCH_BUSY` on saturation) set via `[plugins.desktop.dispatch]` (`timeout_s`, `max_queue`) in `jac.toml`. If the in-process backend fails to boot, the host shows a deterministic native error page instead of navigating to a silently-broken app, and `dispatch()` returns a `NATIVE_BOOT_ERROR` envelope rather than hanging. Native error envelopes follow a production error contract (`{code, message, request_id}`): every error carries a `request_id` for log correlation, and full tracebacks go to stderr only -- echoed into `error.details` solely when debug is enabled via `[plugins.desktop.dispatch].debug` (or the `JAC_DESKTOP_DEBUG` env var). Until jac-scale is bundled into the fused runtime, `sv` dispatch resolves jac-scale from a portable runtime seam (`JAC_DESKTOP_SCALE_SRC` / `JAC_DESKTOP_DEPS`), with no build-machine state baked in.
- **Feature: Native desktop IPC bridge for walker/function calls**: `__doWalkerFetch` and `__doFuncFetch` in the `cl` runtime route through `globalThis.__jac_invoke` when `__JAC_NATIVE__` is set (the Jac-native desktop shell), carrying the frozen `{kind, name, node, fields, token}` request and replaying HTTP status/body semantics locally, including 401 → re-login for walkers.
