- **Feature: bundled native `hashlib`/`hmac` over a static libcrypto floor, plus whole-module native imports (#6978 Phase 1)**: `import hashlib` / `import hmac` now resolve on the native (`nacompile`/JIT) pathway to bundled `na_stdlib` Mechanism-F modules, which are thin pure-Jac surfaces (`Hash`/`HMAC` objects with `update`/`digest`/`hexdigest`/`copy`, the per-algorithm constructors, and `hmac.new`) over OpenSSL's `libcrypto`, reached through an FFI floor (`_hashlib_native.na.jac`). Coverage is the always-built OpenSSL digests `md5`/`sha1`/`sha224`/`sha256`/`sha384`/`sha512` plus `sha3_256`/`sha3_512`; every `hexdigest` is held byte-identical to CPython by a tri-backend equivalence test, and an `nacompile`d binary statically links the bundled `libcrypto.a` (no `libcrypto.so` dependency). This also adds general whole-module imports of bundled `.na.jac` stdlib modules on the native pathway: `import hashlib; hashlib.sha256(...)` now links the module and routes the module-qualified call to its function exactly like the selective `import from hashlib { sha256 }` form (previously only the selective form and a hardcoded `math`/`os`/`sys`/`time`/`random`/`struct` list were accepted). Supporting native-codegen fixes land alongside: a type-correct default-argument fallback (a bare `sha256()` no longer emits an invalid `i32 0` for a `bytes` default, E5020), `:priv` internal linkage on `hashlib`'s internal `new` so it no longer collides with `hmac`'s public `new` at link time (E5026), and ELF static-archive floor-linking fixes (local-symbol namespacing, `.bss`/NOBITS plus COMMON allocation, a local-GOT region, `atexit`/`__dso_handle`, and per-section base alignment).
