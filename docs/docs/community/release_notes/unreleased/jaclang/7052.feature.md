- **Feature: bundled native `ssl` (verifying TLS client) over a libssl/libcrypto floor (#6978 Phase 1)**: `import ssl` now resolves on the native (`nacompile`/JIT) pathway to a bundled `na_stdlib` module -- a pure-Jac verifying-TLS-client surface (`ssl.na.jac`) over an OpenSSL `libssl`/`libcrypto` FFI floor (`_ssl_native.na.jac`, reusing the same bundled `libcrypto` the `hashlib`/`secrets` floors link). The verification posture is CPython's, never hand-rolled: the context is created with `SSL_VERIFY_PEER`, a CA bundle is loaded (the system / `SSL_CERT_FILE` store by default, or an explicit `cafile`), SNI is sent, and the peer hostname is pinned with `SSL_set1_host`, so the handshake fails closed on an untrusted chain OR a hostname mismatch and the surface raises a catchable `SSLError`. Surface: `SSLError`, `SSLContext`, `create_default_context`, and `SSLContext.wrap_socket(sock, server_hostname=...)` returning an `SSLSocket` with `send`/`sendall`/`recv`/`close`. A new equivalence test (`test_ssl_equivalence`) stands up a loopback HTTPS server with a test-generated self-signed cert and asserts, in CI, that a trusted cert + correct hostname verifies and GETs 200 (congruent with CPython), while an untrusted chain and a wrong hostname each fail closed with `SSLError`. A single `SSLError` type is used (not CPython's `SSLCertVerificationError` subclass) pending native base-class `except` matching.
- **Feature: native `urllib.request.urlopen` reimplemented over the `socket`+`ssl` floors; libcurl floor removed (#6978 Phase 1)**: the bundled `na_stdlib/urllib/request.na.jac` is now a pure-Jac surface over the bundled `socket` and `ssl` modules -- it parses the URL, opens the connection (TLS-wrapped and certificate/hostname-verified for `https`), sends an HTTP/1.0 GET, drains the response, splits headers from body, and follows redirects -- replacing the previous libcurl FFI floor. A `nacompile`d program that uses `urlopen` now links no foreign C beyond libc/libssl/libcrypto (**no `libcurl` dependency**, removing the last non-pbs C dependency and the only user-facing variadic-FFI caller). The `urlopen(url) -> HTTPResponse` surface (`status`/`body_length`/`reason`/`url`/`body`/`header_names`/`header_values`) is preserved, and `test_urllib_equivalence` keeps pinning it congruent with CPython's `urllib.request` against a loopback server for a direct GET and a 301 redirect.
