- **Breaking (internal): one dependency engine — ecosystem providers + `resolve_plan`**: All project dependency management now flows through a single resolution pipeline. Each ecosystem (python built-in, npm from the client plugin) registers an `EcosystemProvider` in one registry; `resolve_plan()` turns `jac.toml` plus capability intents into a typed plan where every dependency carries its origin (`manifest`, `extra:<group>`, `capability:<name>`, `self-heal:client-core`, `target:<t>`), and `jac install`/`add`/`remove`/`update`, eject, and publish all consume it. The new `jac deps` command (with `--json` and `--scale`) prints the fully resolved plan. `jac eject` now materializes the same resolved sets: `frontend/package.json` includes the self-healed client core deps and target-scoped deps the dev bundler uses, and `backend/requirements.txt` sources its conditional pins (SQLAlchemy, python-dotenv, python-multipart, LLM stack) from the central capability catalog via new `data.sql`/`env`/`upload` capabilities, annotated with the capability that pulled each one in. Reading client config no longer writes self-healed deps into `jac.toml` — they are injected at resolve time only. For plugin authors this is a clean break: `DependencyInstaller`, `DependencyResolver`, the dict-based `DependencyType` registry, and the `register_dependency_type` hook are removed in favor of `jaclang.project.providers` and the `register_ecosystem_providers` hook (`jac.toml` syntax and all CLI flags are unchanged).
