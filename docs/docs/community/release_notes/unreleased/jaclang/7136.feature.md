- **na→wasm modules carry their own libc and a versioned host import contract**: compiling `na` code to wasm no longer leaks the native floor's libc surface (`strlen`/`strtod`/`malloc`/`snprintf`/...) as unversioned `env` imports that every hand-written JS shim had to chase by hand (#7048). Pure-compute libc (string.h, ctype, `strtod`/`strtol`, `snprintf` formatting, libm, and an allocator) is compiled from a vendored musl/wasi-libc subset to LLVM bitcode (`zig build vendor-wasm-libc`, bundled in shipped binaries) and linked *into* the module, with `printf` formatting in-module and emitting through a single host `write(fd, ptr, len)` instead of JS-side varargs emulation. What still crosses the boundary is the small, versioned `jac_host1` import module (console write/flush, exit/abort/getenv, file-I/O and os stubs, clocks) declared in one place and enforced by the linker: a floor reference to any symbol that is neither in the ABI nor declared by the app's own FFI is now a build-time error naming the symbol, instead of a runtime `LinkError` in the browser. Client apps get the `jac_host1` object from the new `@jac/wasm_host` runtime library, so shims (e.g. the raylib WebGL shim) shrink to just their app's FFI under `env`; the client build also writes a `<module>.wasm.imports.json` manifest of the exact import surface, and modules gain a `name` section for legible stack traces. A typical module drops from ~12 raw libc `env` imports to 4 `jac_host1` imports; without the vendored bitcode the build degrades to the legacy env-import behavior with a warning.
