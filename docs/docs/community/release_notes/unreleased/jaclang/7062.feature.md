- **Feature: bundled native `bz2` over a bundled `libbz2` floor (#6978 Phase 2)**: `import from bz2 { compress, decompress }` resolves on the native (`nacompile`/JIT) pathway to a bundled `na_stdlib` module -- a thin pure-Jac surface (`bz2.na.jac`) over `libbz2`, reached through an FFI floor (`_bz2_native.na.jac`), exactly as CPython's `bz2` wraps the same library. The BWT engine is never reimplemented. The in-process JIT binds the system `libbz2` by logical name; the AOT `nacompile` pathway consumes jac's shipped `libbz2.a` archive via the generic name-derived floor mechanics staged by #6990. Coverage is the one-shot buffer API: `compress(data, compresslevel=9)` produces a single bzip2 stream byte-identical to CPython's (same default `workFactor`), and `decompress(data)` round-trips a single stream, growing its output buffer on `BZ_OUTBUFF_FULL`. The floor wraps `BZ2_bzBuffToBuffCompress` / `BZ2_bzBuffToBuffDecompress`; note `libbz2`'s one-shot API is 32-bit throughout -- `sourceLen` is a by-value C `unsigned int` (lowered as `u32`, unlike zlib's LP64 `uLong`) and the in/out `destLen` is an `unsigned int*` (a 4-byte cell) -- so both directions reject >4 GiB inputs with a `ValueError`, and a valid stream that expands past the `sourceLen * 1024 + 64 MiB` growth ceiling raises a distinct "decompressed output exceeds the one-shot API limit" error. SCOPE: the one-shot buffer API only -- multi-stream inputs return only the first stream's data (CPython concatenates every stream); incremental `BZ2Compressor` / `BZ2Decompressor` and the file API are out of scope; corrupt input raises `ValueError` where CPython raises `OSError`. A new equivalence fixture (`prim_bz2`, registered with `require=["na"]`) pins the native module congruent with CPython's `bz2` on compressed-stream byte-identity (poly digest + exact length at the default and an explicit level), round-trip, empty input, and the error paths (corrupt input, truncated stream, and out-of-range compresslevels all raise on both backends).
