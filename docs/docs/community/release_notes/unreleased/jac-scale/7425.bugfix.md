- **Fix: `--dry-run` no longer installs deploy tooling or touches the cluster**: `jac start --scale --dry-run` built its deployment target before checking `dry_run`, so every dry run ran `_ensure_deploy_tooling()` and pip-installed the `deploy.k8s` dists into the jac global site -- a mutation of the operator's machine that the flag explicitly promised not to make. `--enable-tls` sat in the same window, so `--dry-run --enable-tls` issued real certificates against the live cluster. `dry_run` is now threaded into `DeploymentTargetFactory.create()`, which neither installs the tooling nor probes for it, and it takes precedence over `--enable-tls`. Because manifest rendering is pure, a dry run now also works on a machine that has never installed the deploy tooling. The flag's help text no longer claims to "touch nothing": a dry run still reads the cluster's node architectures, downloads the pod binary, and builds the client bundle, because the plan is content-addressed and would otherwise not match the deploy it previews -- so the text now states what it does guarantee (nothing applied, no DB provisioned, no TLS issued, no tooling installed).
- **Fix: `deploy.k8s` no longer dies on `No module named 'bcrypt'`**: the ingress deployer imported `bcrypt` at module scope to hash the RedisInsight basic-auth htpasswd entry, but `bcrypt` was declared only in the `serve` capability. Since deploy tooling is auto-installed from the `deploy.k8s` dep set, a deploy would sync its tooling and then immediately crash importing a module that tooling never included. `bcrypt` is now part of `deploy.k8s`, and the import moved to its point of use so importing the deploy target no longer requires it.
