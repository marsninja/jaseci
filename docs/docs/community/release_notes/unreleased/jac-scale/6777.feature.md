- **Feature: Zero-config microservice mode (local and `--scale`), with an optional no-image deploy**: An app that splits itself with `sv import from <module>` / `to sv:` now "just works" with no `[plugins.scale.microservices]` config. `jac start main.jac` auto-detects the services (scanning the entry file's directory; comments and docstrings are ignored so documentation mentioning `sv import` is never miscounted), gives each a default `/api/<name>` route, runs them as local subprocesses behind the gateway on `:8000`, and prints what it detected. `jac start main.jac --scale` does the same for Kubernetes: it auto-detects services, swaps to the microservice target, and propagates the discovered routes to every pod via a `JAC_SV_ROUTES` env var (so the gateway and `get_sv_registry` resolve them even though the in-cluster `jac.toml` has no routes table). The `enabled` flag is now tri-state - `true` opts in explicitly, `false` opts out (stays a monolith), and leaving it unset triggers auto-detection - and `jac scale status/stop/restart/logs/destroy` work in auto-detected mode too. Apps with no `sv import` are unaffected and run exactly as before.

- **Feature: no-image microservice deploy (no Docker build/push), automatic when unconfigured**: `jac start --scale` deploys microservices to Kubernetes without building or pushing an image **whenever no image or registry is configured** (no `[plugins.scale.kubernetes].docker_image_name` / `image_registry` and no `--build`) - so a zero-config app deploys with no Docker setup at all. The local source is tar+gzipped into a ConfigMap (with a ~1 MiB guard), a bootstrap initContainer extracts it and `pip install`s the pinned `jaclang` + `jac-scale` releases into a shared volume, and each service runs on the generic `python:3.12-slim` base. Configuring an image/registry (or passing `--build`) keeps the normal image pipeline; `--no-image` forces the copy-source path regardless. Two warnings are printed (build a real image for production; run `jac setup microservice` to configure for production), and MongoDB/Redis are still provisioned so services share state. Cluster-side image builds (buildkit / buildah) are a tracked follow-up.
